Conceptos básicos
sobre Process Mining

📋
Event log → Cases
Cada transacción (case) contiene una secuencia de eventos: PurchaseAttempt CREATED → SimulateDelivery → DeliveryPlan Synced → Checkout Finalized.
🌿
Variant discovery
Agrupar secuencias similares para detectar happy paths y desviaciones importantes. Permite ver dónde se rompe el flujo ideal.
⏱️
Root-cause y timelines
Identificar el paso exacto donde se produce una invalidación o reintento. Medir latencias entre eventos para detectar cuellos de botella.
⚙️
QPR vs reporting
QPR para análisis (variantes/causa). Exportar KPIs simplificados a PBI / Retool si se decide monitorizar de forma continua.
Notas de ponente Resaltar la diferencia entre análisis (QPR) y monitorización: QPR ideal para comprender variantes y diseñar reglas. Para alertas continuas, exportamos KPIs simplificados.

Construcción del dashboard
en QPR

🧩
Event Types
Vista de frecuencia y distribución de eventos clave como Purchase Attempt, Stock Distribution y Delivery Plan. Sirve para validar cobertura del proceso e identificar las actividades dominantes.
⏳
Case Duration Analysis
Análisis de volumen de casos y porcentaje acumulado por duración. Permite detectar cuellos de botella y aislar casos con tiempos anómalos o excesivos.
🛤️
Top Variations
Identificación de los recorridos más comunes del proceso para entender el journey estándar del cliente y localizar desviaciones relevantes sobre el happy path.
✅
Top Variations by Result
Desglose de variantes exitosas y no exitosas para entender qué secuencias favorecen la conversión y cuáles concentran fallos o abandono.
🎯
Path from Selected Event
Exploración específica desde el evento Stock Distribution Created false para analizar rutas problemáticas, anomalías y puntos donde el flujo se desvía.
🔍
Insights principales
La mayoría de los casos siguen el flujo estándar, pero existe un volumen significativo ligado a Stock Distribution Created false; las duraciones largas evidencian ineficiencias y la variabilidad confirma inestabilidad del proceso.
Notas de ponente Presentar el dashboard como una lectura en capas: cobertura de eventos, duración, variantes más frecuentes, impacto por resultado y análisis focalizado sobre eventos problemáticos para convertir hallazgos en acciones priorizadas.
05

Lecciones aprendidas
y conclusiones

💡
Priorizar por impacto
La alta diversidad de variantes implica que no sirve arreglar solo la más frecuente. Medir impacto en conversión y coste — priorizar donde mayor ROI.
📡
Instrumentación importa
Campos faltantes como device_os dificultan segmentación. Forzar captura mejora significativamente el análisis posterior.
♻️
Data reproducible
Guardar caseIds y trazas facilita debugging cross-team (Delivery / Checkout). Ejemplos concretos compartidos vía Grafana + logs.
Notas de ponente Enfatizar instrumentación (device_os) y conservación de caseId en todos los pasos. Mostrar captura Grafana que evidencia LARGEITEMS.
06

Dificultades
y obstáculos

⚠️
Inconsistencias de DeliveryPlan
Casos donde la simulación devuelve LARGEITEMS y nunca aparece STANDARD en la sincronía. Requiere revisión de Delivery Plan (evidencia Grafana en el hilo).
🔀
OPC / COD edge cases
Interacción entre GET /payment-option y POST /shipping genera incoherencias al editar envío. En discusión dos alternativas técnicas.
❌
Errores 422 / invalid shipping
Peticiones PUT/POST con payloads que ocasionalmente devuelven errores 422. Hay trazas y ejemplos en el hilo para reproducir.
💰
Coste ETLs y warehouse
Preocupación por impacto en costes. Acordado enfoque incremental: minivertical → migración si aporta valor demostrado.
Notas de ponente Para cada obstáculo: llevar un caso reproducible (caseId o captura) y una propuesta de mitigación técnica y de negocio.
07

Acciones
recomendadas

⚡ Quick wins · 0–4 sem.
  • Piloto dedup / throttling para filtrar syncs sin cambios — reduce llamadas DeliveryPlanSync innecesarias.
  • Hacer obligatorio device_os en front para mejorar segmentación.
  • Publicar 3 caseIds reproducibles para demo y debugging.
🗓 Mediano plazo · 1–3 m.
  • Simplificación de variantes en QPR y decisión de KPIs a exportar (% sync sin cambios; % invalid; conversión por device_channel).
  • Coordinar con Delivery Plan: LARGEITEMS vs STANDARD y API/contract para OPC — sprint inicia 27/03.
🔭 Largo plazo · 3+ m.
  • Evaluar migración completa a vertical central si minivertical ha demostrado valor y coste es asumible.
  • Diseñar alertas automáticas a PBI / Retool para monitorización continua de KPIs validados.
Notas de ponente Priorizar las acciones 0–4 semanas como decisiones de la reunión. Acordar propietarios y fechas para mediano plazo. Largo plazo como hoja de ruta sujeta a validación.
